1. Multithreading vs Multiprocessing vs Async IO
------------------------------
Multithreading

Uses multiple threads within one process
Blocked by Python's GIL for CPU bound tasks
Best for I/O bound work (network calls, DB I/O, file I/O)
Multiprocessing

Spawns multiple independent processes
Bypasses the GIL
Best for CPU bound workloads (data processing, ML preprocessing)
Async IO

Single threaded cooperative multitasking
Tasks yield control explicitly using
await
Best for high concurrency I/O operations (APIs, DB calls, messaging)
When to choose:

Many HTTP calls  Async IO
Heavy CPU work  Multiprocessing
Simple concurrent I/O in synchronous apps  Threads


2. Python's GIL limitations and workarounds
------------------------------------------

Limitations:

Only one thread executes Python bytecode at a time
CPU bound operations do not scale across threads
Multithreading rarely speeds CPU-heavy workloads
Workarounds used in real projects:

Use multiprocessing for CPU-bound tasks
Offload computation to C-based libraries (NumPy, Pandas)
Use async operations for I/O-bound tasks
Leverage message queues + background workers (Celery/RQ)
Move CPU-heavy work into microservices written in Go/Node/C++

3. Python memory management: reference counting & GC
--------------------------

Reference Counting:

Every object has a reference counter
When count becomes zero  memory freed immediately
Garbage Collection:

Detects reference cycles, e.g. objects referencing each other
Uses generational GC
Runs periodically
Interaction:

Reference counting handles most objects
GC is a fallback for cycles

5. Deep copy vs Shallow copy
---------------------------------


Shallow copy (
copy.copy
)

Copies only top-level reference
Nested objects remain shared
Deep copy (
copy.deepcopy
)

Recursively copies all nested structures
Matters in production when:

Working with nested configs or templates
Mutating nested structures in multi-threaded logic
Modifying default model schemas

10. Ensuring backward compatibility


Never break existing fields  only add new optional ones
Support multiple versions until consumers migrate
Maintain old behaviors via feature flags
Avoid removing deprecated endpoints too early
Use contract testing for API consumers


11. Idempotent vs Safe HTTP methods


Safe: Never change server state

GET, HEAD, OPTIONS
Idempotent: Multiple identical requests produce same result

PUT (replace resource)
DELETE (delete resource)
PATCH is not guaranteed idempotent
Example:
PUT /user/123 {status: active}

 safe to retry

Operations executed in compiled C loops

Executes bulk operations at once

18. Microservice Role-based Authorization

Embed roles in JWT claims
Use API Gateway for centralized RBAC

19. Refresh tokens

Long-lived tokens for obtaining new access tokens
Store securely (HTTP-only cookie)
Rotate on every use
